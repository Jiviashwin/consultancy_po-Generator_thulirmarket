Deep Dive
Complete Workflow Explained
🗺️ End-to-End Flow
Every step from writing code to running in production — your complete DevOps lifecycle.
👨‍💻
1. Developer Writes Code
You write features in React (frontend) or Node.js/Express (backend). Code is organized in the frontend/ and backend/ folders.
frontend/src/ backend/server.js backend/routes/
📤
2. git push → GitHub
You push code to GitHub. Jenkins detects the push via webhook (or polling) and automatically triggers the CI/CD pipeline.
git push origin main Jenkins Webhook Trigger
🔧
3. Jenkins Runs Pipeline (Jenkinsfile)
Jenkins reads your Jenkinsfile and runs 6 stages. Parallel stages run simultaneously to save time.
📥 Stage 1 — Checkout
Clone repo, compute short Git SHA, set image tag like 42-a1b2c3d
📦 Stage 2 — Install (parallel)
Backend: npm ci and Frontend: npm ci run at the same time
🧪 Stage 3 — Test (parallel)
Backend tests + Frontend build check (npm run build) both run simultaneously
🏗️ Stage 4 — Build (parallel)
Two Docker images built simultaneously: po-generator-backend and po-generator-frontend
🐳
4. Docker Builds Container Images
Each service has its own Dockerfile. Docker packages your app + all dependencies into a portable image tagged with build number + git SHA.
🖥️ Backend Image
Node.js + Express app. Runs on port 5002. Includes /api/health endpoint.
⚛️ Frontend Image
Vite builds React → Nginx serves static files on port 80.
ghcr.io/jiviashwin/po-generator-backend:42-a1b2c3d ghcr.io/jiviashwin/po-generator-frontend:42-a1b2c3d
📦
5. Push to GitHub Container Registry (GHCR)
Jenkins authenticates with GHCR using a Personal Access Token stored as a Jenkins credential. Both images are pushed with two tags: version-specific and :latest.
docker login ghcr.io docker push :tag + :latest docker logout ghcr.io
☸️
6. Kubernetes Deploys & Manages
Kubernetes pulls images from GHCR and runs them in the po-generator namespace. It manages 2 replicas of each service with RollingUpdate strategy — zero downtime!
🔄 Rolling Update
New pods start before old ones terminate. maxUnavailable: 0 ensures no downtime.
❤️ Health Checks
Liveness & Readiness probes check /api/health every 15s to ensure pods are healthy.
📊
7. Monitor & Observe
Jenkins marks build as ✅ SUCCESS or ❌ FAILURE. Kubernetes keeps your app running with auto-restart on crash and maintains desired replica count.
kubectl get pods -n po-generator kubectl logs -f deployment/backend Jenkins Build History
🔧
Jenkins CI/CD Server
Runs at http://localhost:8080 | Reads from Jenkinsfile
⚙️
Pipeline Options
Global configuration
  • Keep last 10 builds (log rotation)
  • 30-minute timeout — auto-fails if exceeded
  • No concurrent builds — prevents race conditions
  • Timestamps on every log line
🔑
Credentials
Stored in Jenkins Credentials Store
  • ghcr-credentials — GitHub username + PAT token
  • kubeconfig — Secret file for kubectl access
  • Never exposed in logs — uses withCredentials{}
🏷️
Image Tagging Strategy
Unique per build
  • Format: BUILD_NUM-GIT_SHA (e.g. 42-a1b2c3d)
  • Also tagged as :latest for easy referencing
  • Enables rollback to any previous version
  • Parallel build + push for speed
📋 Post-Build Actions
Jenkins automatically runs cleanup regardless of success or failure.
✅ On SUCCESS
Prints "✅ Pipeline succeeded! Build #N deployed." Local Docker images are cleaned up to save disk space.
❌ On FAILURE
Prints "❌ Pipeline failed." Local Docker images are still cleaned up. Logs are preserved for debugging.
🐳
Docker Overview
Containerization engine
  • Packages app + OS deps into a portable image
  • 2 separate Dockerfiles — backend and frontend
  • Layer caching speeds up repeated builds
  • Images are pushed to GHCR for central storage
🖥️
Backend Container
Node.js + Express API
  • Port: 5002
  • Health: GET /api/health
  • Env from ConfigMap + Secrets
  • Uploads volume persisted via PVC
⚛️
Frontend Container
React/Vite + Nginx
  • Vite builds → static files
  • Nginx serves on port 80
  • Nginx proxies /api/* → backend
  • Only runs after backend is healthy
🐙 docker-compose.yml — Local Development
For local development/testing, docker-compose starts both containers together with a shared po-network. Production uses Kubernetes instead.
📄 Services
backend: Port 5002, env from .env file, healthcheck every 30s

frontend: Port 80, starts only when backend is healthy
🌐 Network & Volumes
Both services on po-network bridge

uploads_data named volume persists uploaded files between restarts
docker-compose up --build docker-compose down -v docker-compose logs -f backend
☸️ Kubernetes Cluster Architecture
Your app runs in the po-generator namespace inside a Kubernetes cluster (Minikube for local, cloud for production).
☸️ Kubernetes Cluster
🌐
Ingress Controller
Routes external traffic → Frontend (/) and Backend (/api) based on URL path rules
namespace: po-generator
🔷 Backend Deployment 2 replicas
backend-pod-1
backend-pod-2
Image: ghcr.io/jiviashwin/po-generator-backend:latest
CPU: 100m–500m | RAM: 256Mi–512Mi
Strategy: RollingUpdate (maxUnavailable: 0)
🔷 Frontend Deployment 2 replicas
frontend-pod-1
frontend-pod-2
Image: ghcr.io/jiviashwin/po-generator-frontend:latest
Nginx serving static React build
Strategy: RollingUpdate
⚡ Services (Internal DNS)
🔗
backend-service
ClusterIP :5002 → backend pods
🔗
frontend-service
NodePort :80 → frontend pods
🗂️ ConfigMap
Non-sensitive config: NODE_ENV, PORT, FRONTEND_URL injected as env vars into pods
🔐 Secrets
Sensitive: MONGODB_URI, JWT_SECRET, SESSION_SECRET, SMTP_USER, SMTP_PASS — encrypted in K8s
💾 PersistentVolumeClaim (uploads-pvc)
Mounted at /app/uploads in backend pods — persists uploaded files across pod restarts
❤️ Liveness Probe
GET /api/health every 15s. After 3 failures, K8s restarts the container automatically.
✅ Readiness Probe
GET /api/health every 10s. Only pods passing this get traffic. Failing pods are temporarily removed from service.
🔧 Jenkins — Pipeline Control
# Start Jenkins with Docker
$ docker-compose -f jenkins/jenkins-docker-compose.yml up -d
# Check Jenkins logs
$ docker logs -f jenkins
# Trigger build manually via Jenkins UI
# Open http://localhost:8080 → pipeline → Build Now
🐳 Docker — Image Management
# Build backend image manually
$ docker build -t ghcr.io/jiviashwin/po-generator-backend:latest ./backend
# List all images
$ docker images | grep po-generator
# Local dev with docker-compose
$ docker-compose up --build
☸️ Kubernetes — Monitor Pods
# Watch all pods in namespace
$ kubectl get pods -n po-generator -w
# Check deployment rollout status
$ kubectl rollout status deployment/backend -n po-generator
# View pod logs live
$ kubectl logs -f deployment/backend -n po-generator
☸️ Kubernetes — Deploy & Manage
# Apply all K8s manifests
$ kubectl apply -f k8s/
# Force update image (manual deploy)
$ kubectl set image deployment/backend backend=ghcr.io/jiviashwin/po-generator-backend:NEW_TAG -n po-generator
# Rollback to previous version
$ kubectl rollout undo deployment/backend -n po-generator
🔍 Kubernetes — Debugging
# Describe a failing pod
$ kubectl describe pod <pod-name> -n po-generator
# Check services
$ kubectl get svc -n po-generator
# Check configmap + secrets
$ kubectl get cm,secret -n po-generator
🚀 Minikube — Local Cluster
# Start Minikube
$ minikube start
# Open dashboard UI
$ minikube dashboard
# Get service URL
$ minikube service frontend-service -n po-generator --url
Tool Breakdown
The Role of Each Tool
🔧
Jenkins
The Automation Brain
  • Detects code changes via GitHub webhook
  • Orchestrates the entire build pipeline
  • Runs tests and build verification
  • Triggers Docker builds and pushes
  • Deploys to Kubernetes via kubectl
  • Reports success/failure status
🐳
Docker
The Packaging Engine
  • Packages your app into portable containers
  • Eliminates "works on my machine" problems
  • Layers allow fast cached rebuilds
  • Stores images in GHCR registry
  • Enables local dev with docker-compose
  • Feeds images to Kubernetes for deployment
☸️
Kubernetes
The Production Operator
  • Runs containers as Pods at scale
  • Maintains desired replica count (2 each)
  • Zero-downtime rolling updates
  • Auto-restarts unhealthy pods
  • Routes traffic via Services + Ingress
  • Manages secrets and configuration
